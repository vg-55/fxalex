# Plan: Weekly Trend + Wick-Based SL

## What we're fixing

Two remaining gaps in the Set & Forget strategy implementation:

1. **Weekly trend** — `fetchClosesDaily` gives us daily closes for EMA-50, but the strategy requires reading the **Weekly/Daily macro structure** (Higher Highs / Higher Lows for uptrend; Lower Highs / Lower Lows for downtrend). We need weekly candle closes to compute a Weekly EMA-50, and a weekly market structure classifier (HH/HL or LH/LL) to confirm macro bias before ANY signal can fire.

2. **SL behind the actual wick** — currently `sl = aoiLow - ATR_buffer`. The strategy says SL goes **below the structural wick of the rejection candle**, not below the static AOI low. When a pin bar prints, the SL should be `candle.low - small_buffer`, which is a tighter and more technically precise placement.

## Part 2: Weekly Multi-Factor Bias Classifier

### `patterns.ts` additions

Add `weeklyBiasScore(candles: Candle[]): { bias: "Bullish" | "Bearish" | "Ranging"; score: number; reasons: string[] }`:

The weekly bias is computed from **3 independent sub-factors**, each casting a vote:

**Sub-factor A: Swing Structure (HH/HL or LH/LL)**
```
1. Detect swing highs: bar where high > both immediate neighbours
2. Detect swing lows: bar where low < both immediate neighbours
3. Last 2 swing highs: HH if rising, LH if falling
4. Last 2 swing lows: HL if rising, LL if falling
   - HH + HL → +2 Bullish votes
   - LH + LL → +2 Bearish votes
   - Mixed (HH + LL or LH + HL) → +1 each = Ranging
   - Insufficient swing points → 0 votes (skip this sub-factor)
```

**Sub-factor B: Weekly EMA-50 Direction**
```
Compute EMA(10) on weekly closes (10-week EMA ≈ 50 EMA proxy on weekly TF):
  close > weekly EMA → +1 Bullish vote
  close < weekly EMA → +1 Bearish vote
  close within 0.2% of EMA → +0 (neutral/ranging)
```

**Sub-factor C: Weekly Momentum (last 3 weekly candles)**
```
Body direction of last 3 weekly closes:
  2 or 3 bullish bodies → +1 Bullish vote
  2 or 3 bearish bodies → +1 Bearish vote
  Mixed → +0 (neutral)
```

**Final classification:**
```
Tally votes:
  Bullish votes >= 3 → "Bullish" (strong)
  Bearish votes >= 3 → "Bearish" (strong)
  Bullish votes == 2 → "Bullish" (moderate)
  Bearish votes == 2 → "Bearish" (moderate)
  Otherwise → "Ranging"

score = Bullish_votes - Bearish_votes (range -4 to +4, normalised to 0–10 for display)
```

This means weekly bias is never a single-point-of-failure — it requires agreement across structure + EMA + momentum to call a direction.

## Part 3: Wick-Based SL Placement

### `engine.ts` changes

Add `wickSl` optional extra:

```ts
BuildSignalExtras {
  ...existing fields...
  rejectionCandleWick?: number | null;  // low of BUY rejection candle, high of SELL
  weeklyBias?: "Bullish" | "Bearish" | "Ranging";
}
```

SL logic update:
```
If rejectionCandleWick is available AND status will be ACTIVE:
  BUY: sl = rejectionCandleWick - (small_buffer = atr * 0.5)
  SELL: sl = rejectionCandleWick + (small_buffer = atr * 0.5)
  → tighter, more precise, technically behind the actual wick
Else (no confirmation candle yet):
  keep existing: sl = aoiLow/aoiHigh - ATR_buffer (zone-based)
```

Why this matters: the wick-based SL is tighter → better R:R, and it's structurally correct per the strategy.

Weekly bias gates:
```
If weeklyBias === "Ranging": cap aiConfidence at 60, add note
If weeklyBias opposes trade direction (weekly Bearish but BUY signal):
  status demoted to PENDING if ACTIVE
  aiConfidence penalised -10
If weeklyBias confirms trade direction:
  aiConfidence bonus +5 (weekly trend aligned with trade)
```

Also add `weeklyTrend` as a new `SignalFactors` sub-field:
```
weeklyTrend: 0 | 5 | 10
  10 = weekly structure aligns with trade direction (HH/HL for BUY)
   5 = weekly ranging (neutral, no edge)
   0 = weekly structure opposes trade direction
```

Adjust factor max totals:
- Old max: 25+20+20+15+10+10+15(AI) = 115
- New max: 25+20+20+15+10+10+10(weekly)+15(AI) = 125 → normalize to /100 for display

Actually to keep the 100-pt system clean: reduce `rejection` max from 20→15 and `momentum` from 15→10, add `weeklyTrend` 0–15. Same total = 100 base + 15 AI.

Updated factor weights:
```
proximity       0–25  (unchanged)
emaConfluence   0–20  (unchanged)
weeklyTrend     0–15  (NEW: weekly HH/HL structure alignment)
rejection       0–15  (was 20 — wick SL makes it less critical as gate)
momentum        0–10  (was 15)
sessionQuality  0–10  (unchanged)
rrQuality       0–10  (unchanged)
aiBoost         0–15  (unchanged)
─────────────────────
base total      0–105 → capped at 100
```

## Part 4: Scanner Changes

### `scanner.ts` additions

In the enrichment block, for each pair also fetch:
```ts
const [candles, dailyCloses, candles1h, weeklyCandles] = await Promise.all([
  fetchCandles4h(pair, 60),
  fetchClosesDaily(pair),
  fetchCandles1h(pair, 20),
  fetchCandlesWeekly(pair, 30),   // ← NEW: 30 weekly bars
]);
```

Compute weekly structure:
```ts
let weeklyBias: "Bullish" | "Bearish" | "Ranging" = "Ranging";
if (weeklyCandles && weeklyCandles.length >= 12) {
  weeklyBias = weeklyMarketStructure(weeklyCandles);
}
```

Extract wick of most recent rejection candle:
```ts
let rejectionCandleWick: number | null = null;
if (rejectionConfirmed && pe.candles1h) {
  const last = pe.candles1h[pe.candles1h.length - 1];
  const prev = pe.candles1h[pe.candles1h.length - 2];
  // Use the candle that triggered rejection
  const rejCandle = isPinBar(last, fourHBias) ? last : prev;
  rejectionCandleWick = fourHBias === "BUY" ? rejCandle.l : rejCandle.h;
}
```

Pass both to `buildSignal`:
```ts
buildSignal(cfg, quote.price, now, {
  ...existing,
  weeklyBias,
  rejectionCandleWick,
});
```

## Key decisions / tradeoffs

**Weekly EMA vs swing structure?**
Both are implemented: swing structure (HH/HL) is primary, weekly EMA-50 is fallback when fewer than 2 swing points exist. Swing structure is more faithful to the strategy description.

**Wick SL tightness risk?**
A wick-based SL is tighter than zone-based, which improves R:R but increases stop-out risk on re-tests. Mitigation: only use wick SL when `rejectionConfirmed = true` AND `insideAOI = true` (ACTIVE signals). PENDING/WATCHING signals keep the wider zone-based SL.

**No DB migration needed?**
- `weeklyBias` stored in `factors` JSONB (already exists)
- `weeklyTrend` score stored in `factors` JSONB
- No new columns needed

**Scan latency impact?**
Weekly candles are cached for 4 hours. First fetch per pair adds ~200ms (parallel with other fetches). Subsequent scans: 0ms overhead.
